## بخش ۱: مشکلات امنیتی بحرانی (بحرانی 🔴)

### 1.1 - عدم وجود محافظت CSRF
**شدت:** بحرانی  
**فایل‌ها:** تمام فرم‌ها و controllers  
**توضیح:** هیچ توکن CSRF در فرم‌ها تولید و اعتبارسنجی نمی‌شود. مهاجم می‌تواند با ساخت یک صفحه مخرب، عملیات را از طرف کاربر لاگین انجام دهد (حذف معامله، تغییر تنظیمات، ارسال پیامک).  
**راه‌حل:** پیاده‌سازی توکن CSRF در Session و اعتبارسنجی در تمام POST requests.

### 1.2 - فایل install.php قابل دسترسی عمومی
**شدت:** بحرانی  
**فایل:** `crm/install.php`, `public/install.php`  
**توضیح:** فایل‌های نصب پس از نصب اولیه قابل دسترسی هستند. مهاجم می‌تواند دیتابیس را ریست کند یا داده‌ها را پاک کند.  
**راه‌حل:** پس از نصب، فایل حذف شود یا بررسی `installed` flag اضافه شود.

### 1.3 - فایل run_repair.php قابل دسترسی عمومی
**شدت:** بحرانی  
**فایل:** `crm/run_repair.php`  
**توضیح:** این فایل در مسیر عمومی قرار دارد و می‌تواند ساختار دیتابیس را تغییر دهد.  
**راه‌حل:** حذف فایل یا انتقال به مسیر غیرقابل دسترس وب.

### 1.4 - عدم تنظیم HttpOnly و Secure برای Session
**شدت:** بالا  
**فایل:** `crm/core/Session.php`  
**توضیح:** تنظیمات session cookie شامل `httponly` و `secure` و `samesite` نیست. مهاجم می‌تواند از طریق XSS به session دسترسی پیدا کند.  
**راه‌حل:** تنظیم `session.cookie_httponly = 1`, `session.cookie_secure = 1`, `session.cookie_samesite = Strict`.

### 1.5 - عدم وجود Rate Limiting در لاگین
**شدت:** بالا  
**فایل:** `crm/controllers/AuthController.php`  
**توضیح:** هیچ محدودیتی برای تلاش‌های ناموفق ورود وجود ندارد. مهاجم می‌تواند brute-force انجام دهد.  
**راه‌حل:** محدود کردن تلاش‌های ورود (مثلاً ۵ تلاش در ۱۵ دقیقه + قفل موقت حساب).

### 1.6 - خروجی raw JSON بدون اعتبارسنجی دسترسی
**شدت:** بالا  
**فایل:** `crm/controllers/DealController.php` (method `getData`)  
**توضیح:** متد `getData()` بدون بررسی لاگین یا دسترسی، اطلاعات معامله را برمی‌گرداند. هر کسی می‌تواند `/deals/data/{id}` را فراخوانی کند.  
**راه‌حل:** اضافه کردن `Auth::requireAuth()` و بررسی دسترسی.

## بخش ۲: مشکلات امنیتی متوسط (متوسط 🟡)

### 2.1 - عدم فیلتر ورودی‌ها در بعضی متدها
**فایل‌ها:** چندین controller  
**توضیح:** در بعضی متدها مثل `update()` در ContactController، ورودی‌ها مستقیماً در آرایه قرار می‌گیرند بدون `trim()` یا اعتبارسنجی.  
**راه‌حل:** فیلتراسیون یکپارچه ورودی‌ها.

### 2.2 - نمایش پیام خطای دیتابیس در حالت Debug
**فایل:** `crm/config/app.php` (`APP_DEBUG = true`)  
**توضیح:** در حالت production، پیام‌های خطای دیتابیس ممکن است اطلاعات حساس (نام جدول، ساختار) را لو بدهد.  
**راه‌حل:** غیرفعال کردن debug در production.

### 2.3 - عدم Hash کردن API Token‌ها
**فایل:** `crm/.env` و جدول `settings`  
**توضیح:** توکن‌های API (Zibal، IPPanel) به صورت plain text ذخیره می‌شوند.  
**راه‌حل:** رمزنگاری توکن‌ها قبل از ذخیره‌سازی.

### 2.4 - عدم بررسی مالکیت در بعضی عملیات SMS
**فایل:** `crm/controllers/SmsController.php`  
**توضیح:** هر کاربری با دسترسی SMS می‌تواند به هر شماره‌ای پیامک ارسال کند بدون بررسی اینکه شماره متعلق به مخاطب خودش باشد.  
**راه‌حل:** بررسی مالکیت مخاطب قبل از ارسال.

### 2.5 - شمارشگر فال‌بک (Fallback) برای session
**فایل:** `crm/core/Session.php`  
**توضیح:** اگر `session_start()` قبلاً فراخوانی شده باشد، ممکن است خطای `session already started` ایجاد شود.  
**راه‌حل:** بررسی `session_status()` قبل از `session_start()`.

## بخش ۳: مشکلات عملکردی CRM (متوسط 🟡)

### 3.1 - عدم وجود احراز هویت دو مرحله‌ای (2FA)
**توضیح:** سیستم فقط با username/password کار می‌کند. برای CRM مالی، 2FA ضروری است.

### 3.2 - عدم وجود سیستم لاگین لاک (Login Lock)
**توضیح:** اگر کاربر چندین بار رمز اشتباه وارد کند، حساب قفل نمی‌شود.

### 3.3 - عدم وجود فراموشی رمز عبور
**توضیح:** اگر کاربر رمز خود را فراموش کند، فقط ادمین می‌تواند رمز را ریست کند.

### 3.4 - عدم وجود سیستم اعلان (Notification)
**توضیح:** هیچ سیستم اعلان داخلی برای یادآوری فعالیت‌ها، پیگیری‌ها و deadline‌ها وجود ندارد.

### 3.5 - عدم وجود Webhook یا API خارجی
**توضیح:** سیستم API REST ندارد. اتصال به سایر نرم‌افزارها ممکن نیست.

### 3.6 - عدم وجود Export داده‌ها
**توضیح:** امکان خروجی Excel/CSV از معاملات، مخاطبان و گزارشات وجود ندارد.

### 3.7 - عدم وجود Import داده‌ها
**توضیح:** امکان ورود داده‌ها از فایل Excel وجود ندارد.

### 3.8 - عدم وجود بکاپ خودکار
**توضیح:** هیچ سیستم بکاپ‌گیری خودکار از دیتابیس وجود ندارد.

## بخش ۵: مشکلات فنی و کدنویسی

### 5.1 - عدم وجود Namespace صحیح برای Helperها
**فایل:** `crm/core/JDate.php`  
**توضیح:** کلاس JDate namespace ندارد و ممکن است conflict ایجاد کند.

### 5.2 - عدم وجود Error Handler یکپارچه
**توضیح:** خطاهای PHP به صورت raw نمایش داده می‌شوند. نیاز به custom error handler داریم.

### 5.3 - عدم وجود سیستم Cache
**توضیح:** تمام queryها هر بار اجرا می‌شوند. برای performance، نیاز به Redis/File cache داریم.

### 5.4 - عدم وجود Validation یکپارچه
**توضیح:** اعتبارسنجی ورودی‌ها در هر controller تکرار می‌شود. نیاز به سیستم validation متمرکز داریم.
